# v0.3.x research item: cold-cache file-hash interaction

**Status:** PARTIALLY RESOLVED 2026-05-27T04:47Z — the headline gap
was methodology artifact; one residual finding (tiny-file dispatch
cost) remains open.
**Filed:** 2026-05-27 by river5 (Mick's bench-consultant role).
**Trigger:** [benchmarker Task 1 v0.1 (2026-05-27T02:29Z)](../HANDOVER.md) —
river5-aesni-v15 lost to blake3 (1.89×) and meowhash (2.73×) at T=1 on
the Dropbox 198 GB cold-cache corpus.

## Resolution summary (2026-05-27T04:47Z)

benchmarker re-ran with hash-bench v0.2.0 (StackHasher in stream
mode; `--mode oneshot` available as a fairness check). All 24 cells:

| Cell                                | river5       | blake3       | meowhash     |
|-------------------------------------|--------------|--------------|--------------|
| Dropbox 198 GB cold-cache T=1       | 0.336 GiB/s  | 0.351 GiB/s  | 0.343 GiB/s  |
| Dropbox 198 GB cold-cache T=32      | ~0.48 GiB/s  | ~0.48 GiB/s  | ~0.48 GiB/s  |

All three hashers within ±5% across every cell. The 1.89× / 2.73×
v0.1.0 gap was **100% methodology artifact** — `Hasher` (heap-alloc
ctx per file) vs `StackHasher` (in-place ctx) on cold-cache
short-file workload.

For river5 specifically, v0.2.0 was 3.46× faster than v0.1.0 on
Dropbox 198 GB T=1 (0.336 / 0.097). The *only* code change was
`river5::Hasher::new()` → `river5::StackHasher::new()` in the stream
path. So on this workload, each `Hasher::new()` was costing roughly
7 ms per file vs near-zero for StackHasher (assuming ~200K files; the
true file count and per-file overhead need profile data to pin down).

### One residual finding

`river5-stream-t1` on the synth200gb tiny-files corpus is 35% slower
than `river5-oneshot-t1` for that cell. blake3 and meowhash don't
show this pattern. **Per-chunk dispatch cost exceeds hash compute
below ~1 KB** for river5 specifically. This is *not* the original
question (which is resolved) — it's a new finding the v2 bench surfaced.

Hypothesis H1 (per-update dispatch overhead) below is now partially
validated: at ~1 KB file size, the per-call setup work in
`river5_update()` is large enough to dominate the per-byte hash cost.

This is the open piece of the doc. Original question CLOSED; new
question OPEN.

### Lesson for me on per-file overhead estimates

My 02:30Z math said `~200ns × 400K files = 80 ms total Hasher::new
overhead = 0.004% of wall time = doesn't explain the gap`. Actual
per-file `Hasher::new` cost on cold-cache short-file workload was
roughly 7 ms — **4 orders of magnitude** larger than my estimate.

The 200 ns figure was probably right for a hot-cache microbench
(plain `malloc` + zero-init); the real-workload cost includes
first-touch page faults on freshly-allocated heap memory, cache
misses on the new ctx's AES round-key data on the first
`update()` call, and possibly TLB pressure from many short-lived
allocations. None of those show up in a `bench_alloc_only`
microbench.

**Don't extrapolate per-file overhead from microbench rules of
thumb when the workload is cold-cache short-file.** Measure on a
representative corpus. Filed as a feedback memory.

## The original question (RESOLVED — methodology artifact)

**Why does river5 specifically lose to blake3 at T=1 on real-workload
cold-cache file-hash, when both are doing the same outer loop
(`open → read 1 MiB → update → finalize`)?**

**Answer (2026-05-27T04:47Z):** v0.1.0 harness used `Hasher` (heap-
alloc ctx). On cold-cache short-file workload, the per-file
`Hasher::new()` cost is large (~7 ms — see lesson above) because of
first-touch page faults + cache misses on the freshly-allocated ctx
+ AES round-key data. Switching to `StackHasher` (in-place ctx)
closes the gap entirely. Original v0.1.0 result was measuring
"river5's heap-alloc ctx" vs "blake3's stack ctx" vs "meowhash's
one-shot-from-Vec", not raw hash throughput.

## The new question (OPEN)

**Why is `river5-stream-t1` 35% slower than `river5-oneshot-t1` on
sub-1 KB files? Why don't blake3/meowhash show the same pattern at
that file size?**

Per-chunk dispatch cost in `river5_update()` exceeds the hash compute
itself below ~1 KB. The v0.2.0 bench surfaced this on the
synth200gb tiny-file corpus. blake3 and meowhash don't show the
crossover at the same file size, so it's river5-specific dispatch
overhead, not a general property of streaming hashers.

This matters because:
1. **Real corpora have small files.** A 'mostly small files' workload
   (npm node_modules, git refs, AppData, etc.) would hit this
   crossover on most files.
2. **One-shot is an obvious workaround for callers who know their
   files are small** (`river5::hash(&buf)` instead of
   `StackHasher::new() + update() + finalize()`). But the streaming
   path should ideally not be 35% slower on tiny files — that's a
   fixable defect in the streaming dispatch.

This is the hypothesis H1 from the original question's list (per-update
dispatch overhead). Partially validated already by the v0.2.0 outlier
data; need a profile run to pin down which specific dispatch step is
the cost (vtable load, ctx pointer reload, tail buffering for
sub-block tails, etc.).

## Why this is worth tracking (not just dropping)

At sd's actual workload (T=32 parallel), all three hashers converge to
within noise — the question doesn't matter for shipped perf today. Two
reasons it's worth keeping the question alive:

1. **Future CPU-bound expansion of sd.** If chromaprint-FFI (engine
   issue #113) or similar v0.3.x perf levers land, CPU-bound phases of
   sd grow. Single-thread hash perf may become relevant downstream.
2. **river5 as a standalone library.** Beyond sd's specific workload,
   understanding *why* a hash interacts poorly with cold-cache file IO
   has standalone value. It could surface an optimization that also
   benefits the hot-cache memory-bandwidth path; or expose a fixable
   per-call setup cost that doesn't show up in microbenches.

## Hypotheses

Listed roughly by plausibility — none verified yet.

### H1 — Per-update dispatch overhead

`river5_update()` (C-side FFI) has fixed per-call work: load vtable,
load ctx pointer, set up registers, copy small tail to internal buf
if not block-aligned. blake3's `update()` does similar work but
possibly less. At ~1 MiB chunks this is amortized over ~1 M bytes per
call, so per-call cost is small. **But** if file sizes in the Dropbox
corpus are small (avg KB-MB, many sub-1 MiB files), there's exactly
one update() call per file with a partial-block tail — fixed overhead
per file.

**Falsifier:** at fixed file size + same hashed-bytes total,
river5's per-call wall time should plateau. If it stays roughly flat
across a sweep of file sizes, dispatch overhead isn't the issue. If
it scales inversely with file size (slower per byte on smaller files),
dispatch overhead IS the issue.

### H2 — Cache miss pattern on AES round-key access

river5's AES-NI path loads the round-key schedule per block. If the
round-key memory is being evicted between files (cold-cache scenario),
each file pays the load cost. blake3 doesn't use AES round keys (it's
ARX-based — `add-rotate-xor`); meowhash uses AES round keys but
possibly with a different access pattern that's cache-friendlier.

**Falsifier:** `perf stat -e LLC-load-misses,L1-dcache-load-misses`
against each hasher on the same corpus. If river5 has materially
more LLC misses per byte than blake3, this is plausibly the cause.

### H3 — Read-buffer dispatch sub-block split

river5's `update()` splits the input into 128-byte AES-block
iterations + a tail. For each iteration there's bookkeeping
(counter increments, lane index updates). At 1 MiB chunks this is
~8 K block iterations per call — fine. But if the bench is using a
smaller buffer or the file is < 1 MiB, the iteration count drops and
the *per-block* fixed cost (function call overhead from the
non-inlined `target` attribute in the AES-NI source) becomes
proportionally larger.

**Falsifier:** rerun the bench with `--bufsize 65536` and
`--bufsize 4194304`. If river5's relative gap to blake3 changes
materially, buffer-size interaction is real.

### H4 — Prefetch interference

The OS pre-fetcher reads ahead on sequential file reads. river5's
inner loop has different memory access patterns than blake3's; one
may interfere more with the pre-fetcher's heuristic than the other.

**Falsifier:** `posix_fadvise(POSIX_FADV_SEQUENTIAL)` before the
read loop, re-bench. If river5's gap closes, the pre-fetcher was
the issue.

### H5 — Open-file syscall noise

Both hashers do the same `File::open()`, but per-file open cost
varies with filesystem state. If the v0.1.0 bench happened to
sequence river5 first (warming cache for blake3 + meowhash) or had
some other ordering artifact, the gap is methodology not algorithm.

**Falsifier:** v2 re-run with randomised hasher order between
runs + several iterations. If river5's gap shifts with ordering,
this is partly methodology not algorithm.

## Proposed investigation method (revised after v0.2.0 results)

Order — cheapest to most expensive. Step 1 done; steps 2-5 below
are revised to target the *new* question (tiny-file stream dispatch
overhead) instead of the resolved one.

1. ✅ **(v0.2.0 re-run.)** DONE 2026-05-27T04:47Z. Confirmed the
   Dropbox cold-cache gap was a methodology artifact (StackHasher
   closes it entirely).
2. **File-size sweep on a controlled corpus.** Generate corpora of
   {64 B, 256 B, 1 KB, 4 KB, 16 KB, 64 KB, 256 KB, 1 MiB, 16 MiB}
   files (uniform within each corpus, say 100 K files per bucket
   except large sizes), run all 3 hashers × stream/oneshot × T=1.
   Identify the exact size where river5-stream loses to river5-
   oneshot. Confirm blake3 and meowhash don't show the same
   crossover or show it at a different size. Tests H1 + H3.
3. **`perf record` + flamegraph on river5-stream below the crossover
   size.** Identify which specific step in `river5_update()` is
   the dispatch cost (vtable load, ctx pointer reload via
   `ctx_ptr()`, sub-block tail buffering, etc). Should take <30 min
   to set up + run on a representative sub-corpus.
4. **Targeted fix experiment.** Probable fixes:
    - Inline `river5_update()`'s fast path when input is shorter than
      one AES block (128 B for v15) — skip the streaming machinery
      entirely, just buffer into the ctx tail.
    - Hoist `ctx_ptr()` re-derivation out of the per-call path —
      add a `Hot` wrapper that captures a stable pointer for the
      lifetime of an `update + finalize` pair (safe because we own
      `&mut self` across both).
    - Pre-compute the per-block scramble constants once at
      `StackHasher::new` rather than per-update.
5. **Re-bench at step 2's crossover sizes after each fix.** Iterate
   until crossover disappears or until cost-of-fix exceeds
   value-of-fix.

## Success criteria for closing this item

Original-question criteria are MET (v0.2.0 already achieves
parity-or-tie on cold-cache file-hash with StackHasher).

For the *residual* tiny-file dispatch finding, either:

- **(a)** `river5-stream-t1` and `river5-oneshot-t1` match within
  10% across all file sizes ≥ 64 B. Then the streaming path is
  competitive at all sizes; users don't need to special-case
  tiny-file workloads.
- **(b)** The `stream`-vs-`oneshot` crossover size is documented
  precisely and callers are advised to use `river5::hash(&buf)`
  when files are known to be below that size. Then the item
  closes with a doc update (river5 README) rather than a code
  change, and (a) becomes a future v0.4.x consideration if
  workload demand surfaces.

Same as before: (a) is the strong outcome but (b) is acceptable.
